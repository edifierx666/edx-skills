# 代码理解型 skill 设计文档

## 背景

当前仓库里已经有面向渲染、图表、终端输出等场景的 skills，但缺少一个专门约束“如何理解代码 / 如何解释项目逻辑”的 skill。

用户给出的规则不是在补充某个知识点，而是在纠正一种常见失败模式：

- 一上来先堆文件路径、组件名、hook 名、调用栈。
- 直接顺着源码结构讲，而不是先建立心理模型。
- 没有先盘点核心状态、关键数据结构和接口载荷。
- 把 UI 交互中间态、派生值、接口参数混在一起解释。
- 讲完后不给下钻选项，导致用户只能继续被动追问。

因此，这个 skill 的本质不是“提供代码阅读技巧”，而是把“代码理解任务”强制拉回到“先建心理模型，再讲业务主链”的解释协议。

## 已确认的产品决策

- 这是一个新的独立 skill，不并入 debug、plan 或 implementation 类 skill。
- 覆盖范围是“讲代码逻辑 + 查看项目 / 模块”，而不是只解释单个函数。
- 它是“先理解再动手”的前置 skill；后续如果进入实现、调试、规划，应切换到对应 skill。
- 首轮解释必须严格遵守固定框架；后续追问允许转成局部下钻。
- 如果用户明确要求先定位代码，可以先给 1-2 句极简心智模型，再给路径或调用链，不需要展开完整五步。
- 遇到“看下这个项目”“帮我分析这个模块”“这个页面怎么跑起来的”这类理解型请求时，自动触发。

## 目标

1. 把“理解项目 / 模块 / 页面 / 业务流”的回答方式统一为心理模型驱动。
2. 让首轮解释优先回答“核心状态是什么、为什么这样流转”，而不是“代码在哪”。
3. 为用户建立稳定、可复用的阅读入口：概览 -> 状态盘点 -> 业务流 -> 流程链 -> 下钻。
4. 在必须贴代码时，强制把每个状态 / 变量 / 字段解释清楚，降低理解门槛。
5. 让 skill 本身具备抗偷懒能力，显式拦截“先堆定位”“顺着文件扫”的默认回答习惯。

## 非目标

- 不负责修 bug。
- 不负责架构设计或实现计划。
- 不负责直接写功能代码。
- 不负责替代 AST / grep / LSP 等检索工具。
- 不把所有技术问答都纳入本 skill；如果只是极小的定位问题或纯 API 查询，不强制展开完整解释协议。

## skill 定位

这个 skill 应属于“纪律型 + 解释协议型” skill。

它解决的不是“Claude 不知道怎么读代码”，而是“Claude 在读代码时默认按自己熟悉的顺序输出，顺序对用户不友好”。

因此正文必须以硬约束为主，而不是散文化建议：

- 什么场景必须触发。
- 首轮回答禁止做什么。
- 必须先输出哪些结构。
- 何时允许降级成简版。
- 代码片段解释的最小粒度是什么。

## 建议 skill 名称与 frontmatter

### 建议名称

`understanding-code-logic`

原因：

- 能同时覆盖“看项目 / 模块 / 页面 / 代码逻辑”。
- 属于动名式表达，符合 skill 命名习惯。
- 不会和 debug、planning、implementation 直接混淆。

### 建议 description

```yaml
name: understanding-code-logic
description: Use when understanding a codebase, module, page, or business flow before making changes, or when the user asks to explain how code or project logic works.
```

要求：

- 以 `Use when` 开头。
- 只描述触发条件和使用场景，不概括五步流程。
- 要覆盖自动触发语义：看项目、看模块、看页面、看业务流、解释代码逻辑。

## 触发范围

应自动触发的典型请求：

- “看下这个项目”
- “帮我分析这个模块”
- “读一下这段代码”
- “这个页面是怎么跑起来的”
- “先看看整体逻辑”
- “解释这段业务流程”
- “这个接口参数是怎么拼出来的”

不应作为主 skill 触发的场景：

- 纯调试：如“为什么报错”“帮我定位 bug”
- 纯实现：如“帮我加个功能”“直接改代码”
- 纯规划：如“怎么设计这个模块”
- 纯定位：如“这个函数在哪个文件”，且用户没有理解诉求

## 回答协议

### 首轮固定结构

首轮解释必须按以下顺序组织：

1. **一眼看懂（Quick Overview）**
2. **盘点核心状态与数据结构**
3. **大白话业务流**
4. **极简泳道图 / 流程链**
5. **按需下钻选项**

skill 需要明确：

- 这不是推荐顺序，而是默认强制顺序。
- 在没有完成“核心状态 / 数据结构”盘点之前，不允许先罗列路径、文件、函数分布。
- “按需下钻选项”是首轮解释的收口动作，不可省略。

### 明确定位请求的降级协议

当用户明确要求定位时，允许走简版：

1. 先用 1-2 句给出极简心智模型。
2. 立即给出路径、模块位置、调用链或行号。
3. 不展开完整五步。

这条规则用于兼容用户的显式指令，避免 skill 与用户目标冲突。

## 核心状态 / 数据结构盘点协议

这是 skill 的核心执行部分，必须写清为固定格式。

### 分类要求

状态与字段必须按用途分组，例如：

- UI 渲染与交互类
- 派生计算类
- 核心业务逻辑类
- 接口请求与载荷类
- 外部依赖 / 参考辅料

### 字段头格式

统一使用：

`字段名：大白话含义 [参考：短标识]`

禁止直接贴大段对象定义来代替解释。

### 字段补充信息格式

每个字段下补充信息按固定顺序单独列出，且不写“来源：”“首次赋值：”这类 key：

- 说明该值从何而来
- 说明首次赋值的依据或默认值
- 说明何时被修改（若有必要）

### 引用收敛

字段解释里不反复写长路径、长接口地址。
所有依赖、接口、工具函数统一沉到底部“参考辅料列表”，字段头里只保留短标识。

### 状态解耦

必须明确区分：

- 前端交互中间态
- 派生计算值
- 最终接口载荷

即使它们之间同步，也不能混讲。

## 代码片段解释协议

当必须输出代码片段，或用户要求解释复杂黑盒代码时：

- 每一个状态 / 变量 / 字段都必须解释。
- 包括嵌套对象里的底层字段。
- 解释必须逐行追加在代码行尾，使用 `// ←`。
- 禁止把多个字段合并成一句笼统说明。

skill 里应明确给出一正一反两个小例子：

- 反例：多个字段共用一条模糊注释。
- 正例：每个字段单独注释。

## 红线约束

skill 需要把以下行为写成明确禁止项：

1. 首轮先堆文件路径、行号、调用栈。
2. 在没讲清核心状态 / 数据结构前，先讲“代码在哪”。
3. 用“顺着文件看”“从入口扫一遍”代替结构化说明。
4. 把 UI 交互态、派生值、接口载荷混在一起。
5. 输出代码但遗漏部分变量 / 字段解释。
6. 讲完整体逻辑后不提供下钻选项。
7. 把未经验证的猜测分支混入主线业务流。

## 基线失败模式

基于无-skill 子代理测试，skill 必须显式覆盖以下失败模式：

- 文件路径先行，心智模型后置。
- 模块名、hook 名、props、useEffect 依赖堆砌成“技术名词墙”。
- 只描述函数调用顺序，不描述状态如何流转。
- 缺少核心对象、状态容器、请求载荷的盘点。
- 只会“按文件顺序看”，不会“按业务主链讲”。
- 没有解释为什么这样设计、这样流转。
- 没有明确的继续下钻选项。
- 一旦贴代码，就容易漏掉逐字段解释。

## skill 正文建议结构

建议 `SKILL.md` 采用以下结构：

1. Frontmatter
2. Overview
3. When to use
4. Hard rules
5. Required response flow
6. State extraction protocol
7. Code explanation protocol
8. Red flags / failure modes
9. Minimal examples

其中：

- `When to use` 负责触发条件。
- `Hard rules` 负责红线。
- `Required response flow` 负责五步协议和定位请求降级协议。
- `State extraction protocol` 负责字段盘点格式。
- `Code explanation protocol` 负责 `// ←` 逐行解释。
- `Red flags / failure modes` 负责防偷懒和防走样。

## 交付范围

第一版交付建议控制在最小范围：

- `skills/understanding-code-logic/SKILL.md`

可选项：

- 如测试中发现需要补充固定示例，再增加 `examples/basic.md`
- 不默认创建 supporting reference 文件，避免过早拆分

## 测试策略

### RED 基线

已完成一轮无-skill 基线测试，确认默认失败模式成立。

### GREEN 验证

实现 skill 后，至少需要验证以下场景：

1. 用户说“看下这个项目”时，能自动触发并先给心智模型。
2. 用户说“帮我分析这个模块”时，会先做状态盘点而不是先给路径。
3. 用户说“这个页面怎么跑起来的”时，会给业务流和流程链。
4. 用户说“直接告诉我在哪个文件”时，会先给 1-2 句极简模型，再定位。
5. 用户要求解释代码片段时，会对每个变量 / 字段做逐行 `// ←` 注释。

### REFACTOR 验证

重点观察是否还有以下新借口：

- “用户只是随便问一下，不需要这么重”
- “先给路径再解释更高效”
- “我已经概括了，不需要盘点状态”
- “这段代码太长，没必要每个字段都注释”

如果出现，应回写到 `Red flags / failure modes` 中。

## 验收标准

1. 首轮解释默认遵守固定五步结构。
2. 自动触发范围覆盖项目 / 模块 / 页面 / 业务流理解场景。
3. 明确定位请求能够走“极简模型 + 定位”降级协议。
4. 状态盘点格式稳定，能区分 UI 中间态、派生值、接口载荷。
5. 代码解释场景下，每个变量 / 字段都有单独 `// ←` 注释。
6. skill 文案本身能显式拦截基线测试中暴露的失败模式。
7. 第一版不与 debug、implementation、planning 类 skill 产生职责混淆。

## 实施提示

实现时应优先保持 skill 本体短而硬：

- description 只写触发条件。
- 正文只保留真正会改变行为的约束。
- 示例只保留最关键的正反例，不扩写成教程。

如果后续验证发现 Claude 仍会偷懒，优先补“红线 + 失败模式 + 反例”，而不是盲目加长解释。
