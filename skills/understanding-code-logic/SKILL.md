# Understanding Code Logic

## Core Positioning

默认假设用户对底层代码实现、复杂变量流转和中间态完全不熟。你的首要任务是帮用户建立清晰的心理模型，而不是充当人肉代码阅读器。

第一目标不是“把代码按顺序读一遍”，而是先让用户理解：这个模块本质上在解决什么问题、围绕哪些核心状态流转、为什么最后会得到当前结果。

如果第一轮回复还在堆文件路径、行号、调用栈、hook 名，说明你已经偏了。先讲核心状态和业务主链，再讲代码位置。

## Overview

设定用户对底层代码实现和复杂变量逻辑完全不熟。你的首要任务是帮用户建立清晰的心理模型，而不是充当人肉代码阅读器。

第一轮如果还在堆文件路径、行号、调用栈、hook 名，说明你已经偏了。先讲核心状态和业务主链，再讲代码位置。

## When to Use

Use when the user wants to understand a project, module, page, business flow, state flow, payload assembly, or a complex block of code before making changes.

典型请求：

- "看下这个项目"
- "帮我分析这个模块"
- "读一下这段代码"
- "这个页面是怎么跑起来的"
- "先看看整体逻辑"
- "解释这段业务流程"
- "这个接口参数是怎么拼出来的"

Do not use for:

- debugging
- root cause analysis
- writing or changing code
- architecture planning
- 纯定位且没有理解意图的问题

## Red Lines

- 第一轮回复中，绝对禁止堆砌具体文件路径、行号或复杂参数名调用栈。
- 没讲清楚“核心状态 / 数据结构是什么”以及“为什么这么做”之前，严禁罗列“代码在哪”。
- 讲解流程时，严禁把 UI 交互中间态、接口返回数据、派生值、最终请求载荷混为一谈。
- 严禁用“我先扫一下文件”“我先顺着目录看”代替真正的业务分析。
- 只要贴代码，就必须解释每一个状态、变量、字段；不允许遗漏。
- 第一轮讲解结束后，必须停下并把控制权交还给用户。

## 强制回复框架（五步法）

### 📊 一眼看懂 (Quick Overview)

在讲具体细节前，先给一个极简全局视角，让用户能在 10 秒内抓住这个模块本质上在干什么。

这一段只回答三件事：

- 这个模块核心在处理什么问题
- 它围绕什么状态或对象流转
- 它最终驱动什么结果

不要在这里先讲文件路径、行号、组件树或调用栈。

### 第一步：大白话业务流 (High-Level Concept)

用最通俗的语言，把核心业务流概括成 1-5 句话。

优先讲：

- 谁触发了这条链路
- 哪些关键状态发生了变化
- 哪些模块消费了这些状态
- 用户最终看到什么结果

不要把这一段写成函数名、hook 名、生命周期名的串联。

示例：

“本质上就是：用户点击时间轴 -> 触发钩子提取时间数据，并更新到全局共享的 `riskTimelineContext` 对象中 -> 地图和图表因为监听到这个对象变了，各自去拉取新数据。”

### 第二步：极简泳道图 / 流程链 (Simplified Data Flow)

使用纯文本画出极简数据流。

要求：

- 只写核心组件名、动作和涉及的关键状态 / 结构
- 绝不写具体文件路径
- 只保留主链路，不把旁支和猜测节点混进来

示例：

```text
[时间轴组件] 点击节点
  → 触发回调，组装出包含 `fctTimeNode` 的数据
    → [TimeRange Hook] 接收数据，生成并更新全局状态 (`riskTimelineContext`)
      → [地图组件] & [各统计组件] 监听到该状态变化
        → 提取状态里的 `timeRange` 等字段，调用各自 API 刷新视图
```

### 第三步：盘点核心状态与数据结构 (Identify Core States & Data Structures)

在讲完业务主线和极简流程链后，再回头盘点支撑这条链路的关键状态变量、关键对象内部字段、接口返回数据、请求载荷和共享上下文。

必须做到：

1. **强制分类聚类**
   - 按用途分组，例如：**UI 渲染与交互类**、**接口返回数据类**、**派生计算类**、**核心业务逻辑类**、**接口提交与载荷类**。
   - 分类标题本身使用单独一行的粗体文本，**不要**在左侧再加 `-`、`*`、数字序号或箭头。
   - 用层级缩进清楚展示，不能混成一坨列表。
   - **UI 渲染与交互类不是“其中一类”，而是 UI 层全量总表。**
   - 只要变量、状态或方法满足下面任意一条，就必须出现在 **UI 渲染与交互类** 中：
     - 在模板 / JSX / 渲染函数里直接出现
     - 决定 UI 展示内容、显示状态、禁用状态、显隐状态、激活状态、选中状态、展开状态、排序状态、筛选状态、分页状态
     - 是用户可直接触发的事件方法，如点击、输入、切换、打开、关闭、提交、重置、分页、重试、保存
     - 是接口返回后直接用于渲染、回显、列表展示、表单回填、按钮态控制的数据变量
   - 这一条允许与其他分类冗余；**凡是 UI 上看得到、点得到、触发得到、依赖得到的变量 / 状态 / 方法，都要先收进 UI 渲染与交互类，宁可重复，不可漏项。**
   - **UI 渲染与交互类中的条目不能只平铺罗列。只要某个 UI 状态、UI 变量或 UI 方法依赖其他状态，就必须用缩进树把依赖链展开出来。只列结果、不列依赖，视为不完整。**

2. **字段头部格式固定**
   - 每个字段统一使用：`字段名：大白话含义 [参考：短标识]`
   - 不要直接贴整段对象定义来代替解释。
   - 对于 **UI 渲染与交互类**，如果某个条目依赖其他状态、变量、方法或接口返回值，必须继续向下缩进展开依赖树。

3. **带可见缩进的分层结构**
   - 分类下面的**第一级字段条目**不要再使用有序序号，统一使用 ASCII 箭头：`->`。
   - **只有字段详情层**才使用：`a.`、`b.`、`c.`。
   - 固定语义为：`a.` = 来源，`b.` = 首次赋值，`c.` = 变更时机。
   - 实际输出时不要再重复“来源：”“首次赋值：”“变更时机：”这几个字，直接写内容。
   - **不要把分类下的字段本体写成 `a. / b. / c.`。** 那会和详情层混在一起，层级会塌。
   - **不要依赖行首普通半角空格做缩进。聊天渲染会折叠这些空格，视觉上看起来就像没缩进。**
   - **必须使用可保留的缩进字符。默认使用全角空格 `　` 作为前导缩进。**
   - 一级字段本身使用 `->`；字段下的 `a. / b. / c.` 前统一补两个全角空格；依赖树里的子节点继续用数字序号，子节点详情再使用 `a. / b. / c.`。
   - 例如：
     - `-> active：当前是否被选中 [参考：无]`
     - `　　a. 前端组件内部状态`
     - `　　b. 默认为 false`
     - `　　c. 用户点击激活时触发改变`
   - 如果某字段没有明确的变更时机，也保留 `c.`，并直接写“当前链路中无单独变更动作”或“随上游状态变化同步更新”。
   - 对于依赖树，继续向下缩进，并在每个依赖节点下同样遵守“节点用数字序号，节点详情用 `a. / b. / c.`”这条规则。
   - 这里追求的是文本层级清晰，不要求渲染器把 `a.` 识别成原生有序列表；重点是让缩进字符在最终聊天界面里真的显示出来。

4. **引用收敛**
   - 不要在字段说明里反复写长路径、长依赖链或长接口地址。
   - 统一沉到底部，整理成“参考辅料列表”。
   - 字段头里只放短标识，例如 `[接口A]`、`[依赖A]`。

5. **状态解耦精准识别**
   - 严禁遗漏 UI 交互中间态。
   - 必须明确区分：
     - 用于前端交互控制的变量
     - 从接口获取并进入前端状态 / 上下文 / store 的数据变量
     - 派生出来的计算值
     - 最终发给接口、提交给服务器的载荷参数
   - 即使它们同步，也必须拆开讲。
   - “接口返回数据变量”和“提交给服务器的数据变量”必须分开建类或分开列项，禁止混成同一组“接口数据”。
   - **同一个变量如果既参与 UI 展示，又属于接口返回数据或派生值，必须在 UI 渲染与交互类里再列一次。不要因为它已经出现在别的分类里，就从 UI 分类中省略。**

格式示例：

**UI 渲染与交互类**

  -> active：当前是否被选中 [参考：无]
  　　a. 前端组件内部状态
  　　b. 默认为 false
  　　c. 用户点击激活时触发改变

  -> displayTitle：界面标题文本 [参考：接口A]
  　　a. 由当前模式和时间轴展示数据共同计算得到
  　　b. 默认模式与首次接口返回共同决定
  　　c. 用户切换模式或接口返回新数据时同步更新
  　　依赖树：
  　　　 1. params.source：当前界面展示与交互使用的预报模式 [参考：接口A]
  　　　　　a. 模式按钮选择的值
  　　　　　b. 默认取全局配置项的首个模式
  　　　　　c. 用户切换模式标签时同步更新
  　　　 2. timelineData：时间轴展示数据 [参考：接口A]
  　　　　　a. 由时间轴接口返回后直接用于 UI 展示
  　　　　　b. 接口首次响应结果
  　　　　　c. 重新请求接口后被覆盖更新

  -> handleTabChange：模式切换方法 [参考：无]
  　　a. 绑定在 UI 事件上的交互方法
  　　b. 组件初始化时定义
  　　c. 用户点击标签时触发

**接口返回数据类**

  -> timelineData：接口返回的时间轴数据 [参考：接口A]
  　　a. 由时间轴接口返回
  　　b. 接口响应结果
  　　c. 重新请求接口后被覆盖更新

**接口提交与载荷类**

  -> submitPayload.source：最终提交给服务器的预报模式 [参考：接口B]
  　　a. 由当前选中模式整理得到
  　　b. 当前 UI 选择对应的模式值
  　　c. 用户修改筛选条件或点击提交前会重新组装

**参考辅料列表：**

[接口A] 预报时间轴接口，请求地址: /warn/statfcttimelinewarninfo
[依赖A] 时间工具库，文件: utils/time.js -> formatTime

### 第四步：提供按需下钻的选项 (On-Demand Details)

完成上述解释后，必须停下来，把控制权交还给用户。

结尾至少给 2-3 个明确的下钻选项，让用户决定下一步看哪一层，而不是继续自顾自展开源码导览。

示例：

“以上是整体的数据流向和核心结构。您想先深入了解哪一块的具体代码？比如：1. `riskTimelineContext` 是通过什么函数拼装出来的？2. 地图组件具体是怎么监听并触发重绘的？”

## 变量与代码解释铁律

当不可避免需要输出代码，或代码涉及复杂逻辑、核心数据结构时：

- **每一个**状态、变量、字段都必须解释，不能遗漏。
- 包括嵌套对象里的最底层子字段。
- 必须在对应代码行尾使用 `// ←` 追加大白话解释。
- 严禁把多个字段合并成一句模糊说明。

反例：

```ts
preFromFileId, windFromFileId, temFromFileId // 缺少对每一个独立字段的解释
```

正例：

```ts
preFromFileId,  // ← 降水模型的文件 ID
windFromFileId, // ← 风场模型的文件 ID
temFromFileId,  // ← 温度模型的文件 ID
```

## 输出检查清单

在发送首轮解释前，检查：

- 是否先给了 Quick Overview，而不是先报路径
- 是否先讲了大白话业务流和极简流程链，再回头盘点核心状态 / 数据结构
- 是否把 UI 上出现、影响 UI、绑定到 UI 的变量 / 状态 / 方法全部收进了 **UI 渲染与交互类**
- 是否把接口返回后直接参与 UI 展示的数据也重复收进了 **UI 渲染与交互类**
- 是否给 UI 类中的关键条目补了缩进树依赖链，而不是只平铺结果
- 是否明确拆开了接口返回数据、UI 中间态、派生值、最终 payload / 提交给服务器的数据
- 是否先讲了“为什么这样流转”，再讲代码位置
- 是否给了纯文本极简流程链
- 是否在结尾给了下钻选项
- 如果贴了代码，是否每个变量 / 字段都有 `// ←` 注释

## Failure Modes To Block

遇到这些默认输出，要主动拦住：

- 一上来先讲文件，再讲逻辑
- 先堆 hook、props、生命周期，再讲系统到底在干什么
- 跳过大白话业务流和极简流程链，直接钻进状态清单
- 只讲函数调用顺序，不讲状态如何流转
- 把 UI 渲染与交互类写成局部示例，而不是 UI 全量总表
- 只列 UI 结果状态，不列它依赖的上游状态树
- 跳过 UI 可见变量 / 方法盘点、接口返回数据盘点或 payload 盘点
- 按源码顺序讲，而不是按业务主链讲
- 讲完后不给下钻选项
- 贴了代码，但字段解释不完整

## Location-First Override

如果用户明确要求“先告诉我在哪个文件 / 哪条调用链”，可以降级处理：

1. 先用 1-2 句给出极简心智模型。
2. 再给路径、调用链或行号。
3. 停下来，等用户决定要不要继续展开五步法。

即使在这种场景下，也不要完全跳过心智模型。
