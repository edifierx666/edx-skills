# Understanding Code Logic

## Overview

默认把用户当作第一次接触这段源码的人：不知道目录结构，不知道组件之间怎么协作，不知道状态从哪里来，也不知道函数名背后的业务含义。

你的任务不是“顺着源码念一遍”，而是把复杂实现压缩成一份第一次看就能读懂的说明书：先建立心智模型，再补充代码证据。

输出不仅要正确，还要美观易读。首轮回复必须像一页结构化说明，而不是一段口头转述：要有清晰标题、短段落、极简流程图、分类后的状态清单，以及最后的下钻选项。

## What Good Looks Like

- 用户在 10 秒内知道这个模块本质上在解决什么问题。
- 用户能看懂：谁触发链路、哪些状态变化、这些状态被谁消费、最后落成什么结果。
- 用户能区分：UI 中间态、接口返回、派生值、最终提交载荷或副作用输出。
- 用户会拿到少量关键代码落点作为证据，但不会一上来被文件路径、调用栈和参数名淹没。
- 整份输出有视觉层级，能直接拿去当阅读笔记，而不是“口头说说”。

## When to Use

Use when the user wants to understand a project, page, module, state flow, request payload assembly, business chain, or a complex block of code before making changes.

典型请求：

- "看下这个项目"
- "帮我分析这个模块"
- "读一下这段代码"
- "这个页面是怎么跑起来的"
- "先看看整体逻辑"
- "解释这段业务流程"
- "这个接口参数是怎么拼出来的"

Do not use for:

- debugging
- root cause analysis
- writing or changing code
- architecture planning
- 纯定位且没有理解意图的问题

## Core Principle

先让用户明白“系统为什么会这样流转”，再让用户看到“代码写在什么地方”。

如果第一轮回复还在堆文件路径、行号、hook 名、组件树、调用栈，说明你已经偏了。用户对源码是一无所知的；在他脑子里，文件名和函数名本身没有解释力。

## Red Lines

- 默认假设用户对源码、目录、框架术语一无所知。第一次提到 `hook`、`context`、`store`、`selector`、`DTO`、`adapter` 这类术语时，必须立刻补一句大白话解释。
- 第一轮回复中，绝对禁止一上来先报文件路径、行号、调用栈或复杂参数名。代码位置只能放在后面的“代码落点”区，或者用户明确要求“先看位置”时再提前给。
- 严禁把解释写成一整段口语化长文。必须输出带标题、空行、流程块、分类树的可读结构。
- 没讲清楚“核心状态 / 数据结构是什么”以及“为什么这样流转”之前，严禁开始按源码顺序导览。
- 讲解流程时，严禁把 UI 交互中间态、接口返回数据、派生值、最终请求载荷或副作用结果混为一谈。
- 严禁用“我先扫一下文件”“我先顺着目录看”代替真正的业务分析。
- 只要贴代码，就必须解释每一个状态、变量、字段；不允许遗漏。
- 第一轮讲解结束后，必须停下，并把控制权交还给用户。

## 输出排版契约

为了让解释真正“可读”，而不是“听起来像解释过了”，首轮回复必须遵守下面的版式要求：

- 必须有分段标题。推荐顺序：`一眼看懂`、`大白话业务流`、`流程链`、`核心状态与数据结构`、`代码落点`、`可继续下钻的方向`。
- 每个自然段尽量短，避免出现一大段 6-10 行的密集文字墙。
- 流程必须放进纯文本代码块，用极简链路展示，不要把流程藏进普通段落里。
- 分类盘点必须有清晰视觉层级；不要把所有状态揉成一个扁平列表。
- 标识符、字段名、方法名、对象名、接口短标识统一放进反引号中，帮助用户快速扫读。
- 当输出“依赖树”或“字段详情”时，必须使用可保留的缩进字符，不要依赖普通半角空格。
- 代码落点必须简短，起“证据”作用，而不是重新开始一轮源码导览。

## 术语翻译规则

用户对源码是零基础，所以术语第一次出现时，必须立即翻译成人话。

示例：

- `context`（跨组件共享状态的容器）
- `store`（集中存放全局状态的地方）
- `selector`（从全局状态里取值的函数）
- `payload`（最终发给后端的参数对象）
- `adapter`（把外部数据整理成内部统一结构的转换层）

如果术语不翻译，用户看到的就不是解释，而是另一层黑话。

## 强制回复框架（六步法）

### 第一步：📊 一眼看懂 (Quick Overview)

在讲具体细节前，先给一个极简全局视角，让用户能在 10 秒内抓住这个模块本质上在干什么。

这一段只回答三件事：

- 这个模块核心在处理什么问题
- 它围绕什么状态或对象流转
- 它最终驱动什么结果

不要在这里先讲文件路径、行号、组件树或调用栈。

### 第二步：🧠 大白话业务流 (High-Level Concept)

用最通俗的语言，把核心业务流概括成 1-5 句话。

优先讲：

- 谁触发了这条链路
- 哪些关键状态发生了变化
- 哪些模块消费了这些状态
- 用户最终看到什么结果

不要把这一段写成函数名、hook 名、生命周期名的串联。

示例：

“本质上就是：用户点击时间轴 -> 页面拿到新的时间范围 -> 共享状态被更新 -> 地图和图表都监听这个共享状态，所以会各自重新请求数据并刷新展示。”

### 第三步：🗺️ 极简泳道图 / 流程链 (Simplified Data Flow)

使用纯文本画出极简数据流。

要求：

- 只写核心组件名、动作和涉及的关键状态 / 结构
- 绝不写具体文件路径
- 只保留主链路，不把旁支和猜测节点混进来
- 图里的名词要尽量能让零基础用户看懂；必要时用括号补一句人话

示例：

```text
[时间轴组件] 用户点击节点
  → 触发回调，产出新的时间范围数据
    → [共享时间状态] 接收并保存这组时间范围
      → [地图组件] 和 [统计组件] 监听到时间变化
        → 各自用新的时间范围重新请求接口并刷新页面
```

### 第四步：🧩 盘点核心状态与数据结构 (Identify Core States & Data Structures)

在讲完业务主线和极简流程链后，再回头盘点支撑这条链路的关键状态变量、关键对象内部字段、接口返回数据、请求载荷和共享上下文。

这一步的目标不是“列名词”，而是帮用户把脑子里的对象层次搭起来。

必须做到：

1. **强制分类聚类**
   - 按用途分组，例如：**UI 渲染与交互类**、**外部输入类**、**接口返回数据类**、**派生计算类**、**核心业务逻辑类**、**接口提交与载荷类**。
   - 如果当前场景是页面、组件、表单、图表、筛选器、弹窗、列表、时间轴这类可交互界面，那么 **UI 渲染与交互类** 是必选项，不得省略。
   - 如果当前场景不是 UI（例如后端任务、脚本、数据处理），可以把分类改成更贴合的结构，例如：**输入与触发类**、**运行中状态类**、**外部返回数据类**、**派生结果类**、**输出与副作用类**。
   - 分类标题本身使用单独一行的粗体文本，**不要**在左侧再加 `-`、`*`、数字序号或箭头。
   - 用层级缩进清楚展示，不能混成一坨列表。

2. **UI 渲染与交互类必须是全量总表**
   - 只要变量、状态或方法满足下面任意一条，就必须出现在 **UI 渲染与交互类** 中：
     - 在模板 / JSX / 渲染函数里直接出现
     - 决定 UI 展示内容、显示状态、禁用状态、显隐状态、激活状态、选中状态、展开状态、排序状态、筛选状态、分页状态
     - 是用户可直接触发的事件方法，如点击、输入、切换、打开、关闭、提交、重置、分页、重试、保存
     - 是接口返回后直接用于渲染、回显、列表展示、表单回填、按钮态控制的数据变量
   - 这一条允许与其他分类冗余；**凡是 UI 上看得到、点得到、触发得到、依赖得到的变量 / 状态 / 方法，都要先收进 UI 渲染与交互类，宁可重复，不可漏项。**

3. **字段头部格式固定**
   - 每个字段统一使用：`字段名：大白话含义 [参考：短标识]`
   - 不要直接贴整段对象定义来代替解释。
   - 对于 **UI 渲染与交互类**，如果某个条目依赖其他状态、变量、方法或接口返回值，必须继续向下缩进展开依赖树。

4. **带可见缩进的分层结构**
   - 分类下面的**第一级字段条目**不要再使用有序序号，统一使用 ASCII 箭头：`->`。
   - **只有字段详情层**才使用：`a.`、`b.`、`c.`。
   - 固定语义为：`a.` = 来源，`b.` = 首次赋值，`c.` = 变更时机。
   - 实际输出时不要再重复“来源：”“首次赋值：”“变更时机：”这几个字，直接写内容。
   - **不要依赖行首普通半角空格做缩进。聊天渲染会折叠这些空格，视觉上看起来就像没缩进。**
   - **必须使用可保留的缩进字符。默认使用全角空格 `　` 作为前导缩进。**
   - 一级字段本身使用 `->`；字段下的 `a. / b. / c.` 前统一补两个全角空格；依赖树里的子节点继续用数字序号，子节点详情再使用 `a. / b. / c.`。
   - 如果某字段没有明确的变更时机，也保留 `c.`，并直接写“当前链路中无单独变更动作”或“随上游状态变化同步更新”。

5. **状态解耦精准识别**
   - 严禁遗漏 UI 交互中间态。
   - 必须明确区分：
     - 用于前端交互控制的变量
     - 从接口获取并进入前端状态 / 上下文 / store 的数据变量
     - 派生出来的计算值
     - 最终发给接口、提交给服务器的数据变量
   - 即使它们同步，也必须拆开讲。
   - “接口返回数据变量”和“提交给服务器的数据变量”必须分开建类或分开列项，禁止混成同一组“接口数据”。
   - **同一个变量如果既参与 UI 展示，又属于接口返回数据或派生值，必须在 UI 渲染与交互类里再列一次。不要因为它已经出现在别的分类里，就从 UI 分类中省略。**

6. **输出要服务阅读，不是服务凑字数**
   - 选最关键的状态、对象和字段，不要机械穷举与当前主链无关的局部细节。
   - 但是，只要你已经贴出一段代码，里面出现的字段就必须解释完整，不能挑着讲。

格式示例：

**UI 渲染与交互类**

-> `active`：当前是否被选中 [参考：无]
　　a. 前端组件内部状态
　　b. 默认为 `false`
　　c. 用户点击激活时触发改变

-> `displayTitle`：界面顶部显示的标题文本 [参考：接口A]
　　a. 由当前模式和时间轴展示数据共同计算得到
　　b. 默认模式与首次接口返回共同决定
　　c. 用户切换模式或接口返回新数据时同步更新
　　依赖树：
　　　1. `params.source`：当前页面展示与交互使用的模式值 [参考：接口A]
　　　　a. 模式按钮选择的值
　　　　b. 默认取全局配置项的首个模式
　　　　c. 用户切换模式标签时同步更新
　　　2. `timelineData`：时间轴展示数据 [参考：接口A]
　　　　a. 由时间轴接口返回后直接用于 UI 展示
　　　　b. 接口首次响应结果
　　　　c. 重新请求接口后被覆盖更新

-> `handleTabChange`：模式切换方法 [参考：无]
　　a. 绑定在 UI 事件上的交互方法
　　b. 组件初始化时定义
　　c. 用户点击标签时触发

**接口返回数据类**

-> `timelineData`：接口返回的时间轴数据 [参考：接口A]
　　a. 由时间轴接口返回
　　b. 接口响应结果
　　c. 重新请求接口后被覆盖更新

**接口提交与载荷类**

-> `submitPayload.source`：最终提交给服务器的模式值 [参考：接口B]
　　a. 由当前选中模式整理得到
　　b. 当前 UI 选择对应的模式值
　　c. 用户修改筛选条件或点击提交前会重新组装

**参考辅料列表：**

`[接口A]` 预报时间轴接口，请求地址: `/warn/statfcttimelinewarninfo`
`[依赖A]` 时间工具库，文件: `utils/time.js` -> `formatTime`

### 第五步：📍 代码落点 / 证据 (Code Anchors)

当用户已经建立了心智模型后，可以在后面补一个很短的“代码落点”区，作为证据和继续阅读的入口。

要求：

- 放在业务解释和状态盘点之后，不能抢到前面去。
- 只列最关键的 2-6 个落点，不要把整个目录树都贴出来。
- 每个落点只做一件事：告诉用户“这块代码负责什么”。
- 如果没有被问到，不要在这里继续展开函数调用细节。

推荐格式：

- `src/pages/RiskMap/index.tsx`：页面入口，负责把筛选条件、时间状态和子组件串起来
- `src/hooks/useTimeRange.ts`：接收时间轴点击结果，整理出共享时间范围状态
- `src/components/MapPanel/index.tsx`：监听时间范围变化后重新拉取地图数据

### 第六步：🔎 提供按需下钻的选项 (On-Demand Details)

完成上述解释后，必须停下来，把控制权交还给用户。

结尾至少给 2-3 个明确的下钻选项，让用户决定下一步看哪一层，而不是继续自顾自展开源码导览。

示例：

“以上是整体的数据流向和核心结构。接下来如果继续下钻，通常最有价值的是这几块：1. 共享状态是在哪一步被组装出来的；2. 页面上每个图表分别监听了哪些字段；3. 最终请求载荷是怎么从 UI 选择一步步拼出来的。”

## 首轮输出模板

首轮解释默认按下面版式组织，保证既有心智模型，也有视觉层级：

````md
## 一眼看懂
用 2-3 句话说清：它解决什么问题、围绕什么状态流转、最后驱动什么结果。

## 大白话业务流
按“谁触发 -> 什么变化 -> 谁消费 -> 用户看到什么”写成 1-5 句话。

## 流程链
```text
[触发方]
  → [状态变化]
    → [核心消费者]
      → [最终结果]
```

## 核心状态与数据结构
**UI 渲染与交互类**
...

**接口返回数据类**
...

**派生计算类**
...

**接口提交与载荷类**
...

## 代码落点
- `path/to/file`：负责什么
- `path/to/file`：负责什么

## 可继续下钻的方向
1. ...
2. ...
3. ...
````

## 变量与代码解释铁律

当不可避免需要输出代码，或代码涉及复杂逻辑、核心数据结构时：

- **每一个**状态、变量、字段都必须解释，不能遗漏。
- 包括嵌套对象里的最底层子字段。
- 必须在对应代码行尾使用 `// ←` 追加大白话解释。
- 严禁把多个字段合并成一句模糊说明。

反例：

```ts
preFromFileId, windFromFileId, temFromFileId // 缺少对每一个独立字段的解释
```

正例：

```ts
preFromFileId,  // ← 降水模型的文件 ID
windFromFileId, // ← 风场模型的文件 ID
temFromFileId,  // ← 温度模型的文件 ID
```

## 输出检查清单

在发送首轮解释前，检查：

- 是否先给了 `Quick Overview`，而不是先报路径
- 是否先讲了大白话业务流和极简流程链，再回头盘点核心状态 / 数据结构
- 是否默认把用户当成第一次接触源码的人，而不是默认用户知道框架术语
- 是否在第一次出现术语时补了大白话翻译
- 是否把 UI 上出现、影响 UI、绑定到 UI 的变量 / 状态 / 方法全部收进了 **UI 渲染与交互类**
- 是否把接口返回后直接参与 UI 展示的数据也重复收进了 **UI 渲染与交互类**
- 是否给 UI 类中的关键条目补了缩进树依赖链，而不是只平铺结果
- 是否明确拆开了接口返回数据、UI 中间态、派生值、最终 payload / 提交给服务器的数据
- 是否先讲了“为什么这样流转”，再讲代码位置
- 是否给了纯文本极简流程链
- 是否在业务解释之后补了一个简短的“代码落点”区，而不是完全停留在口头概括
- 是否在结尾给了下钻选项
- 如果贴了代码，是否每个变量 / 字段都有 `// ←` 注释
- 整体版式是否像一页说明书，而不是一段密集口述

## Failure Modes To Block

遇到这些默认输出，要主动拦住：

- 一上来先讲文件，再讲逻辑
- 先堆 hook、props、生命周期，再讲系统到底在干什么
- 跳过大白话业务流和极简流程链，直接钻进状态清单
- 只讲函数调用顺序，不讲状态如何流转
- 把 UI 渲染与交互类写成局部示例，而不是 UI 全量总表
- 只列 UI 结果状态，不列它依赖的上游状态树
- 跳过 UI 可见变量 / 方法盘点、接口返回数据盘点或 payload 盘点
- 按源码顺序讲，而不是按业务主链讲
- 术语不翻译，默认用户知道 `context`、`store`、`payload` 在说什么
- 整段输出没有视觉层级，看起来像一坨口头记录
- 讲完后不给下钻选项
- 贴了代码，但字段解释不完整

## Location-First Override

如果用户明确要求“先告诉我在哪个文件 / 哪条调用链”，可以降级处理：
